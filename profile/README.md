# GitHub 運用ルール

**Project / Issue / コミットメッセージ 作成ルール**　制定日：2026年7月15日

---

## 1. Project 作成ルール

| 項目 | 内容 |
|---|---|
| 作成単位 | 案件（工号）単位で1つのProjectを作成する。 |
| Project名 | 案件名をそのまま使用する。<br>例：「○○会社向け □□機能追加」「○○会社向け ××システム新規開発」 |
| カスタムフィールド | 「工号」フィールドを追加し、案件の管理番号（例：`123456K7890`）を記載する。 |
| 紐づくレポジトリ | 案件に関連する全レポジトリのIssueをProjectに追加する（複数レポジトリ横断可）。 |
| ステータス管理 | 案件がクローズしたら、Projectを「Closed」にする（削除はしない。履歴として残す）。 |

**例**
```
Project名：○○会社向け □□機能追加
工号（カスタムフィールド）：123456K7890
状態：Open（作業中） → 案件終了後 Closed
対象レポジトリ：MOG4、MOG4-batch
```

## 2. Issue 作成ルール

### 2-1. タイトル

- タイトル先頭に工号を `【 】` で囲んで記載する。
- 形式：`【工号】タスク内容`

**例**
```
【123456K7890】△△設定画面を追加
```

### 2-2. ラベル

- 工号ラベル（例：`123456K7890`）を必ず付与する。org／レポジトリを横断して工号検索ができるようにするため。
- 種別ラベル（bug／feature／task 等）もあわせて付与する。

### 2-3. Projectへの追加

- 作成したIssueは、対応する案件のProjectに必ず追加する。

### 2-4. クローズ

- 自動クローズ（コミットメッセージでの `Fixes #123` 等）は使用しない。
- 作業完了を確認のうえ、Issue画面で手動でクローズ操作を行う。

**例**
```
タイトル：【123456K7890】△△設定画面を追加
ラベル：123456K7890, feature
Project：○○会社向け □□機能追加
本文：
　△△設定画面に□□項目を追加する。
　対象：管理画面 > 設定 > △△タブ
```

## 3. コミットメッセージ作成ルール

### 3-1. 基本形式

- 関連Issueを、レポジトリ名を省略せず必ず記載する（同一レポジトリ内であっても省略しない）。全レポジトリが同一Organization内にあるため、`owner/` は付与しない。
- 工号を `【 】` で囲んで記載する（履歴を追いやすくするため）。
- 形式：`repo#Issue番号 【工号】変更内容` ※ `#Issue番号` の後ろは半角スペース必須

**例**
```
MOG4#123 【123456K7890】△△設定画面を追加
```

### 3-2. 自動クローズキーワードの禁止

- `Fixes`、`Closes`、`Resolves` 等のキーワードは使用しない（Issueクローズは手動操作のため）。

### 3-3. ブランチ運用

- 作業は必ずブランチを作成して行う。ブランチ命名規則は既存ルールを踏襲する。
- コード修正はブランチにコミットし、masterへはPull Requestでマージする。masterへの直接pushは行わない。

### 3-4. Pull Request ルール

| 項目 | 内容 |
|---|---|
| PRタイトル | 先頭に工号を `【 】` で記載する。<br>形式：`【工号】対応内容` |
| PR本文 | 関連Issueを `repo#Issue番号` の形式で必須記載する。 |
| マージ先 | master |

**例**
```
PRタイトル：【123456K7890】△△設定画面の追加対応
PR本文：
対応Issue：MOG4#123
△△設定画面に□□項目を追加。動作確認済み。
```
